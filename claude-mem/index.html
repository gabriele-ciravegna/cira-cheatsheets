claude-mem 10.6.3

Memoria persistente per Claude Code. Ogni sessione salvata in SQLite. A SessionStart inietta automaticamente il contesto delle sessioni precedenti.

Quando usarlo: SEMPRE. È il sistema nervoso che fa "ricordare" Claude tra sessioni. Senza, ogni /clear azzera tutto. Già attivo nel sistema CIRA.
NON usarlo se: Vuoi sessioni completamente effimere (rarissimo). O se devi escludere contenuto sensibile (usa tag <private>).

Quick Start

# Install (npm)
npm install -g claude-mem

# Worker HTTP auto-avvio a ogni SessionStart
# Porta: 37777

# Web viewer
open http://localhost:37777

# Cerca nella memoria via skill
Skill('mem-search')

Worker gira in background, SQLite DB in ~/.claude-mem/claude-mem.db. Auto-inject all'avvio Claude Code: l'AI vede subito i progetti + decisioni delle sessioni precedenti.

Come funziona

  1. SessionStart hook avvia worker :37777 se non già up, fetcha memorie correlate al cwd e le inietta come system context.
  2. Durante sessione ogni tool call + user prompt + assistant response indicizzati in SQLite con metadata (timestamp, cwd, session_id, project).
  3. A fine sessione (Stop hook) salva summary + decisioni chiave. Indicizzazione FTS5 per ricerca veloce.
  4. Web viewer :37777 per browse sessioni passate, filtrare per progetto/data, leggere transcript.

CLI / Worker

claude-mem serve
Cosa faAvvia worker HTTP su :37777. Bootstrappato automatic da SessionStart hook.
QuandoVerifica manuale post-blackout: curl :37777/health

Cosa faCerca dentro la memoria a tempo di sessione. Query natural language → trova trascript/decisioni rilevanti.
Esempio"come avevo fixato il bot Telegram a maggio?" → ritrova session di 2 settimane fa.
SQLite diretto
Cosa faPer ingest custom: INSERT raw in ~/.claude-mem/claude-mem.db con singleton sdk_session.
QuandoWorker NON espone REST API per write. Per import esterno fai INSERT SQL diretto (vedi memoria reference_claude_mem_sqlite_schema.md).

Tag privacy

Per escludere contenuto dalla memoria, wrappalo:

<private>Token: sk-ant-... </private>
<private>Email cliente: foo@bar.com</private>

Hook filtra il contenuto tra <private> tag prima di scrivere al DB. Comunque resta nel transcript JSONL Claude Code stesso (~/.claude/projects/) — per zero trace usa env var dedicato + script interattivo.

CIRA Stack

Memoria persistente Claude Code

claude-mem è già attivo a livello GLOBAL su ~/.claude/CLAUDE.md. Il blocco "Active Systems" lo lista come sistema 0. Worker :37777 boot automatic.

CIRA Integration

Memoria file-based parallela

Parallelo a claude-mem: /Users/cira/.claude/projects/-Users-cira/memory/*.md con MEMORY.md index. È auto memory built-in del nostro sistema, separato dal SQLite di claude-mem (diversa granularità).

CIRA Use

Backup mensile

Cron domenica 04:00 mirror ~/.claude-mem/ + memory/ → SAMSUNG SSD CIRA-BACKUP-memclaude/. Vedi memoria project_backup_strategy.md.

Gotchas & Anti-pattern

  • Worker :37777 deve restare up — se sessione Claude Code non parte, verifica lsof -iTCP:37777 e riavvia manuale.
  • Token/secrets in chat finiscono nel DB e nei JSONL — usa SEMPRE script interattivo (es. setup_keychain.py) per inserire credenziali, mai paste in chat.
  • SQLite single-writer: NO scrittura concorrente da più sessioni; usa lock o accoda via worker.
  • Db cresce nel tempo — vacuum periodico: sqlite3 ~/.claude-mem/claude-mem.db 'VACUUM'.
  • Web viewer è pubblico su localhost:37777 — NON espone fuori da loopback ma sniffing locale possibile.
