Superpowers 2.5.0

TDD-first + brainstorming + verification loop. Spec-light, test-heavy collaborative design.

Quando usarlo: Codice critico testabile con coverage 80%+. TDD obbligatorio. Brainstorm design prima di scrivere. Multi-agent review su architetture complesse.
NON usarlo se: Prototipo throwaway, spec ricca esiste già (usa GSD invece), task <30min senza test, frontend bootstrap solo (usa frontend-design)

Quick Start

Pipeline TDD standard:

brainstorming → writing-plans → tdd-red → tdd-green → tdd-refactor → code-review-excellence

Skill invocation pattern: Skill('brainstorming') da REPL Claude Code.

CIRA Examples

CIRA Context

Superpowers vs GSD

GSD: compliance + audit trail heavy. Superpowers: TDD + code quality heavy. Less planning overhead.

Usa GSD quando spec-driven complex project, audit trails needed, stakeholder signoff critical.
Usa Superpowers quando codice critico, test coverage non-negotiable, TDD discipline required, team review collaboration.

CIRA Integration

Path A3 in /costruiamo v2.5.0

Superpowers è attivo come Path A3 (spec-light + TDD enforcement) nel flow /costruiamo. Il layer TDD è embeddable in /gsd:execute-phase per compliance.

Orchestrazione consigliata: skill-router → multi-agent-brainstorming → brainstorming → writing-plans → executing-plans/tdd-cycle → code-review-excellence

CIRA Metrics

Target Coverage & Quality

Coverage: 80% minimo, 85%+ raccomandato, 100% critical path.
Code quality: cyclomatic complexity <10, method <20 lines, class <200 lines.
Test speed: unit <100ms, integration <1s, E2E <5s per scenario.

Stack Integration

  • GSD — Embeddable: superpowers TDD dentro /gsd:execute-phase per tasks. Coverage 80%+ requirement comune
  • Gstack — Complementare: usa TDD layer in /gstack:pair-agent per realtime collaboration
  • Ruflo — Compatibile per UX iteration loop (brainstorm → prototype → test → refactor)
  • Cloud env — Subagent support highly recommended (Sonnet + Haiku parallel execution). Works without but slower
  • GitHub — All commits TDD-backed. PR flows match code-review-excellence
  • Tooling — Jest/pytest/JUnit detection automatic. No framework lock-in

Gotchas & Anti-patterns

  • Skip brainstorming thinking 'too simple' → WRONG. Every feature needs design approval
  • Start code before design doc approved → WRONG. Brainstorm HARD-GATE
  • Skip tdd-red phase → WRONG. Tests MUST fail first defining behavior
  • Refactor while tests failing → WRONG. Only refactor with tests GREEN
  • Use multi-agent-brainstorming for casual ideas → WRONG. Reserved for critical architecture
  • Don't stop at blocker thinking you can force through → WRONG. Ask help immediately
  • Run executing-plans in same session as writing-plans → CAN work but defeats separate-context isolation
  • Review code without understanding requirements → WRONG. Code-review needs context + tests
